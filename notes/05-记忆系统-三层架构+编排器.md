## 一、记忆系统总览

记忆系统让Hermes能跨会话记住东西——你的偏好、环境信息、踩过的坑、做过的事。和技能（程序化记忆：怎么做）不同，记忆是**事实性记忆**（什么是什么）。

Hermes 虽然没有在代码中显式命名 "long_term" / "short_term"，但实现了清晰的**三层记忆分层架构**，由 `MemoryManager` 统一编排。

```
┌─────────────────────────────────────────────────────────────────────┐
│                        记忆系统层级架构                               │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  L3: 长期记忆 (Long-term Memory)  ← 跨会话持久化              │   │
│  │  ┌─────────────────────┐  ┌──────────────────────────────┐   │   │
│  │  │ MEMORY.md/USER.md   │  │  外部 Provider (Honcho/Mem0/  │   │   │
│  │  │ (内置文件记忆)       │  │  Hindsight/Holographic/...)   │   │   │
│  │  └─────────────────────┘  └──────────────────────────────┘   │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                           ▲ 写入                                   │
│                           │                                       │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  L2: 短期/情景记忆 (Episodic Memory)  ← 跨会话历史检索        │   │
│  │  ┌──────────────────────────────────────────────────────┐   │   │
│  │  │  SQLite + FTS5 全文搜索 (session_search 工具)          │   │   │
│  │  │  保存所有会话的完整消息历史                             │   │   │
│  │  └──────────────────────────────────────────────────────┘   │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                           ▲ 来源                                   │
│                           │                                       │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  L1: 工作记忆 (Working Memory)  ← 当前上下文窗口             │   │
│  │  ┌──────────────────────────────────────────────────────┐   │   │
│  │  │  messages[] 列表 (LLM 上下文窗口)                      │   │   │
│  │  │  → ContextCompressor 自动压缩摘要                      │   │   │
│  │  └──────────────────────────────────────────────────────┘   │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                     │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │                 MemoryManager 编排器统一管理                 │   │
│  │   内置Provider始终启用 + 最多一个外部Provider 并行工作         │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**核心设计原则**：
- `MemoryManager` 是唯一入口，统一管理多个记忆后端
- 内置记忆提供者（builtin：本地文件MEMORY.md/USER.md）始终启用
- **最多允许一个外部记忆提供者**（Honcho/Hindsight/Mem0等）同时运行
- 一个提供者出错不会影响其他提供者，记忆系统永远不会把整个Agent搞挂
- 所有慢操作（sync/prefetch/外部API）全部后台异步，绝不阻塞用户

## 二、三层记忆分类详解

### 2.1 L1: 工作记忆（Working Memory）—— 当前上下文窗口

#### 是什么？
就是当前发送给 LLM 的 `messages[]` 数组，是模型**当下能直接"看到"的所有内容**。

| 属性 | 值 |
|------|-----|
| 生命周期 | 单会话内，压缩后以摘要形式保留 |
| 容量 | 模型上下文窗口大小（如 128k/200k/1M tokens） |
| 访问速度 | 即时（直接在prompt里） |
| 持久性 | 会话结束后不直接保留（会进入L2历史） |

#### 管理机制：上下文压缩

**核心文件**：[context_engine.py](file:///e:/github/hermes-agent/agent/context_engine.py) + [context_compressor.py](file:///e:/github/hermes-agent/agent/context_compressor.py)

**压缩触发时机**：
- **预飞压缩**：回合开始前估算token数，超阈值就压缩
- **实时压缩**：对话过程中token数达到阈值时触发

**保护策略**（防止关键信息被压缩掉）：
```python
protect_first_n = 3   # 保护前3条非系统消息（通常是开场指令）
protect_last_n = 6    # 保护最近6条消息（当前对话上下文）
threshold_tokens = 0.75 * context_length  # 75% 时触发压缩
```

**压缩方式**：让 LLM 把旧消息摘要成一段总结文本，替换掉被压缩的消息。

### 2.2 L2: 短期/情景记忆（Episodic Memory）—— 会话历史

#### 是什么？
所有会话的**完整消息历史**持久化在 SQLite 数据库中，可以通过 `session_search` 工具随时检索。

**核心文件**：
- 会话持久化：[hermes_state.py](file:///e:/github/hermes-agent/)
- 检索工具：[session_search_tool.py](file:///e:/github/hermes-agent/tools/session_search_tool.py)

| 属性 | 值 |
|------|-----|
| 生命周期 | 持久保存，除非手动删除 |
| 容量 | 磁盘空间（几乎无限） |
| 访问方式 | 通过 `session_search` 工具检索，零LLM开销 |
| 索引 | SQLite FTS5 全文搜索引擎 |

#### 三种检索模式

| 模式 | 用途 | 示例 |
|------|------|------|
| **Discovery** | 全文搜索 | `session_search(action="discover", query="上次说的那个bug")` → 返回匹配片段±5条消息上下文+bookend |
| **Scroll** | 滚动查看 | `session_search(action="scroll", session_id="xxx", anchor_message_idx=42, window=10)` → 以某消息为锚点看前后N条 |
| **Browse** | 浏览会话列表 | `session_search(action="browse", limit=20)` → 按时间列出最近会话（标题、预览、时间戳） |

#### 会话谱系（Session Lineage）
通过 `parent_session_id` 跟踪会话关系：
- `/branch` - 从当前会话分支
- `/resume` - 恢复历史会话
- 上下文压缩 - 旧会话归档，新会话继续

#### L3-A: 内置文件记忆（Builtin Memory）

**核心文件**：[memory_tool.py](file:///e:/github/hermes-agent/tools/memory_tool.py)

**两个记忆文件**：

| 文件 | 用途 | 字符上限 | 内容示例 |
|------|------|---------|---------|
| **MEMORY.md** | Agent 的个人笔记 | 默认 2200 字符 | 环境事实、项目约定、工具特性、经验教训 |
| **USER.md** | 用户画像 | 默认 1375 字符 | 姓名、角色、偏好、沟通风格、工作习惯 |

存储位置：`$HERMES_HOME/memories/MEMORY.md` 和 `USER.md`

**条目格式**：
- 条目分隔符：`§`（section sign，章节符号）
- 支持多行条目
- 启动时自动去重（保留首次出现）

```markdown
# Memory Notes
§
项目使用Python 3.11+，用uv管理依赖
§
用户喜欢简洁的回复，不要废话
§
Windows环境下用PowerShell，不要用bash命令
```

##### 🔑 关键设计：冻结快照模式（Frozen Snapshot）

这是一个极其重要的设计！

```
会话启动时：
  ├─ 从磁盘加载 MEMORY.md/USER.md
  ├─ 生成 _system_prompt_snapshot（冻结快照）
  └─ 快照注入到系统提示 → 全程不变！

会话中：
  ├─ 调用 memory 工具写入 → 立即持久化到磁盘
  └─ 但！系统提示中的快照不变！（保证 prefix cache 命中）

下次会话启动：
  └─ 重新加载 → 新的记忆才会出现在系统提示中
```

**为什么这么设计？**
1. **Prefix Cache 友好**：系统提示在整个会话内字节级不变，可以100%命中缓存，省钱又快
2. **写入立即持久化**：不会丢数据
3. **下次会话生效**：新记忆在下次启动时刷新快照

##### 安全与可靠性机制

| 机制 | 说明 |
|------|------|
| **威胁扫描** | 注入系统提示前用 `threat_patterns.py` strict模式扫描，命中的条目标记为 `[BLOCKED: ...]` 占位符，防止记忆投毒 |
| **原子文件写入** | tempfile + `fsync` + `os.replace()`，保证读者总能看到完整文件 |
| **外部漂移检测** | 如果文件被patch/shell/手动编辑导致无法解析，拒绝写入并创建 `.bak.<timestamp>` 备份 |
| **跨平台文件锁** | Unix: fcntl, Windows: msvcrt，防止并发写入损坏 |

**操作API：`memory`工具**
```python
memory(action="add", target="memory", content="项目使用uv管理依赖")
memory(action="replace", target="user", old_substring="喜欢简洁回复", content="用户喜欢详细解释")
memory(action="remove", target="memory", substring="旧约定")
memory(operations=[...])  # 批量原子操作
```

## 三、MemoryManager 编排器模式

**核心文件**：[memory_manager.py](file:///e:/github/hermes-agent/agent/memory_manager.py)

这是整个记忆系统的"大脑"，统一管理所有Provider。

### 3.1 为什么用编排器模式？

在编排器模式出现之前，每个记忆后端的代码散落在run_agent.py里——加一个新的记忆提供者就要改主循环逻辑。MemoryManager把所有记忆相关操作收敛到一个类，run_agent.py只需要和MemoryManager对话，不需要知道背后有几个提供者。

| 问题 | 没有编排器 | MemoryManager编排器 |
|------|-----------|-------------------|
| 主循环代码 | 每个Provider写一遍prefetch/sync，散落在run_agent里 | 调用一次prefetch_all/sync_all搞定 |
| 错误隔离 | Provider抛异常直接崩整个Agent | try/except包住每个Provider，一个挂了其他照常工作 |
| 工具注入 | 每个Provider自己往tools里加schema | 统一normalize_tool_schema、统一去重、统一检查核心工具冲突 |
| 后台线程 | 每个Provider自己管理线程，可能阻塞主循环 | 单worker串行化写入，后台执行永远不阻塞turn完成 |
| 会话切换 | 每个Provider自己监听session_id变化 | 统一分发on_session_switch |
| 技能脚手架污染 | Provider直接拿原始query，把/skill的SKILL.md正文存进记忆 | _strip_skill_scaffolding统一剥离，只存用户真实指令 |

### 3.2 MemoryManager 结构

```python
class MemoryManager:
    def __init__(self):
        self._providers: List[MemoryProvider] = []  # 已注册的提供者列表
        self._tool_to_provider: Dict[str, MemoryProvider] = {}  # 工具名→提供者路由表
        self._has_external: bool = False  # 是否已经注册了外部提供者（防止多个）
        self._sync_executor: Optional[ThreadPoolExecutor] = None  # 后台单worker线程池
```

**关键约束**：
- **内置（builtin）提供者永远第一个注册，永远存在**
- **最多只能有一个外部（非builtin）提供者**——注册第二个直接拒绝并打warning
- 核心工具名（clarify、delegate_task等）保留，记忆提供者不能注册同名工具（内置永远赢）
- 工具名冲突时先注册的赢，后注册的打warning忽略

> 为什么"最多一个外部提供者"？注释原文：*Prevents tool schema bloat and conflicting memory backends.* 多个外部记忆后端会导致：工具列表膨胀、记忆写入冲突、用户不知道哪个记忆源是真相来源。

### 3.3 核心设计原则

| 原则 | 实现方式 |
|------|---------|
| **内置优先 + 单外部可插拔** | 内置文件记忆永远可用，额外可选一个高级Provider |
| **Prefix Cache 友好** | 系统提示会话内冻结，记忆更新不打断缓存 |
| **绝不阻塞用户** | 所有慢操作（sync/prefetch/外部API）全部后台异步 |
| **故障隔离** | 每个Provider独立try/except，一个坏了不影响其他 |
| **写入顺序保证** | 单worker后台线程，turn N先于turn N+1落盘 |
| **Provider极简契约** | ABC接口最小化，新Provider只需实现4个核心方法 |

## 四、可插拔 Provider：MemoryProvider ABC

所有记忆提供者都继承 `agent/memory_provider.py` 里的 `MemoryProvider` ABC，实现统一接口。

### 4.1 必须实现的核心方法（4个）

| 方法 | 作用 |
|------|------|
| `name` (property) | 短标识符："builtin"/"honcho"/"hindsight"/"mem0" |
| `is_available()` | 检查配置/凭证/依赖是否就绪（不做网络调用） |
| `initialize(session_id, **kwargs)` | 会话启动时初始化（建连、建表） |
| `get_tool_schemas()` | 返回暴露给模型的工具schema列表（OpenAI function-calling格式） |

### 4.2 核心生命周期方法（有默认空实现）

| 方法 | 调用时机 |
|------|---------|
| `system_prompt_block()` | 系统提示组装时，返回静态说明文本 |
| `prefetch(query, session_id)` | 每轮对话**前**同步召回，要快（返回缓存） |
| `queue_prefetch(query, session_id)` | 每轮对话**后**为下一轮异步预热缓存 |
| `sync_turn(user, asst, session_id, messages)` | 每轮对话后异步持久化 |
| `handle_tool_call(tool_name, args)` | 处理本Provider的工具调用，返回JSON字符串 |
| `shutdown()` | 清理资源、关闭连接 |

### 4.3 可选钩子（Opt-in Hooks）

不需要就不用实现，默认空操作：

| 钩子 | 触发时机 | 用途 |
|------|---------|------|
| `on_turn_start(turn_number, message, **kwargs)` | 每轮开始 | 计数、定期维护 |
| `on_session_end(messages)` | 真正的会话边界（退出/reset/超时） | 会话级事实提取/摘要 |
| `on_session_switch(new_id, parent, reset, rewound)` | session_id中期轮换（/resume/branch/压缩） | 更新per-session缓存状态 |
| `on_pre_compress(messages)` | 上下文压缩前 | 从即将被压缩的消息中提取重要信息 |
| `on_memory_write(action, target, content, metadata)` | 内置memory工具写入时 | 镜像写入到外部Provider |
| `on_delegation(task, result, child_session_id)` | 子代理完成时 | 父代理接收子代理的观察结果 |
| `get_config_schema()` | `hermes memory setup` 向导 | 定义配置字段 |
| `save_config(values, hermes_home)` | 配置向导完成时 | 写入Provider原生配置文件 |
| `backup_paths()` | `hermes backup` 备份 | 声明HERMES_HOME外的磁盘路径 |

这个ABC定义了记忆提供者的**完整生命周期**——从初始化到关闭，每个时机都有对应的钩子。外部插件只需要继承这个类、实现需要的方法，就能无缝接入。

### 4.4 Provider注册流程

1. 插件放在 `plugins/memory/<name>/` 目录
2. 用户在config.yaml设置 `memory.provider: honcho`
3. Agent启动时加载对应插件，调用 `MemoryManager.add_provider()`
4. add_provider做3件事：
   - 检查是否已经有外部提供者了，有就拒绝
   - 把provider加到 `_providers` 列表
   - 遍历provider.get_tool_schemas()，normalize后注册到 `_tool_to_provider` 路由表，检查核心工具名冲突

## 五、记忆具体工作流

记忆操作分为三个时机：System Prompt构建 → 回合前预取回忆 → 回合后后台写入。

```
┌─────────────────────────────────────────────────────────────────────┐
│ 0. Agent启动时：initialize_all                                      │
│    for provider in providers:                                       │
│       provider.initialize(session_id, hermes_home, platform, ...)  │
│    内置 MemoryStore.load_from_disk() → 生成冻结快照                │
│    （建连、建表、加载数据、启动后台线程）                              │
└──────────────────────────────┬──────────────────────────────────────┘
                               │
                               ▼
┌─────────────────────────────────────────────────────────────────────┐
│ 1. System Prompt构建：build_system_prompt()                         │
│    for provider in providers:                                       │
│       blocks.append(provider.system_prompt_block())                 │
│    内置记忆快照注入系统提示                                          │
│    （把provider的静态说明文字拼进System Prompt，比如"你有记忆功能"）   │
└──────────────────────────────┬──────────────────────────────────────┘
                               │
                               ▼
┌─────────────────────────────────────────────────────────────────────┐
│ 2. 每回合用户发消息 → 回合开始：on_turn_start()                     │
│    provider.on_turn_start(turn_number, user_message, ...)          │
└──────────────────────────────┬──────────────────────────────────────┘
                               │
                               ▼
┌─────────────────────────────────────────────────────────────────────┐
│ 3. 回合前预取回忆：prefetch_all(user_message) 【同步，快！】         │
│    ├─ _strip_skill_scaffolding(query) 剥离技能脚手架                │
│    │  （/skill调用时SKILL.md正文很大，不应该存进记忆）               │
│    └─ for provider in providers:                                   │
│         result = provider.prefetch(clean_query)  ← 返回缓存结果     │
│         parts.append(result)                                       │
│    合并所有provider结果 → sanitize清理已有的围栏标签                │
│    包装成 <memory-context> 围栏块                                   │
│    注入到对话历史中（作为system note），模型看到"这是你的记忆"       │
│                                                                     │
│    为什么prefetch要快？因为它在调用LLM之前同步执行，慢了会增加TTFT。  │
│    provider应该用queue_prefetch后台预取，prefetch只返回缓存。        │
└──────────────────────────────┬──────────────────────────────────────┘
                               │
                               ▼
┌─────────────────────────────────────────────────────────────────────┐
│ 4. LLM处理 → 工具调用 → 回复用户                                    │
│    模型如果主动读写记忆，调用memory相关工具 → handle_tool_call分发    │
│    _tool_to_provider[tool_name] 路由到对应provider                  │
│    内置memory工具写入 → 威胁扫描 → 写审批门 → 文件锁 → 原子写入     │
│    → memory_manager.notify_memory_tool_write() 镜像到外部Provider   │
└──────────────────────────────┬──────────────────────────────────────┘
                               │
                               ▼
┌─────────────────────────────────────────────────────────────────────┐
│ 5. 回合完成后：sync_all(user, asst) 【后台，不阻塞】                 │
│    提交到单worker ThreadPoolExecutor后台执行：                      │
│    ├─ _strip_skill_scaffolding剥离脚手架                             │
│    └─ for provider in providers:                                   │
│          provider.sync_turn(user, asst, messages=...) 写入记忆      │
│                                                                     │
│    为什么后台执行？注释里记录了血的教训：                            │
│    一个配置错的Hindsight daemon阻塞了298秒才失败                     │
│    如果sync_turn是同步的：用户看到回复后Agent还显示"running"几分钟   │
│    用户发新消息触发激进中断，体验极差。                              │
│    后台执行：用户拿到回复就可以继续聊，记忆写入在后台默默完成。       │
└──────────────────────────────┬──────────────────────────────────────┘
                               │
                               ▼
┌─────────────────────────────────────────────────────────────────────┐
│ 6. 回合完成后：queue_prefetch_all(user_message) 【后台】            │
│    让provider为下一回合提前做回忆（比如embedding搜索预计算）          │
│    这样下一回合prefetch()直接拿缓存，不阻塞                         │
└──────────────────────────────┬──────────────────────────────────────┘
                               │
                               ▼
┌─────────────────────────────────────────────────────────────────────┐
│ 会话切换/结束/压缩/子代理时：对应钩子                                │
│  on_session_switch → 更新session_id相关状态                        │
│  on_session_end → 会话结束时做事实提取/总结                         │
│  on_pre_compress → 上下文压缩前提取重要信息存进记忆                 │
│  on_delegation → 父代理观察子代理结果，写入记忆                     │
│                                                                     │
│  注意：非primary上下文不写入记忆                                    │
│  - "primary"：主对话 → 写入                                         │
│  - "subagent"：子代理 → 跳过（临时工作不污染用户记忆）              │
│  - "cron"：定时任务 → 跳过（系统提示词会corrupt用户画像）           │
└─────────────────────────────────────────────────────────────────────┘
```

### 5.1 关键设计：单worker串行化写入

```python
self._sync_executor: Optional[ThreadPoolExecutor] = None  # 单worker！
```

为什么是**单**worker线程池，不是多worker并行？
- 记忆写入必须按顺序：回合N必须在回合N+1之前写入，否则顺序错乱
- 如果用多线程，网络抖动可能导致N+1比N先落库，回忆时顺序乱了
- 一个worker天然串行化，保证写入顺序
- 懒创建：builtin-only路径根本不创建线程池，零额外开销

## 六、🔑 关键机制：记忆围栏与流式Scrubber

记忆注入时有个关键问题：记忆内容是 `<memory-context>...</memory-context>` 围栏包起来的，还有系统提示"This is recalled memory, NOT new user input"。这些是给模型看的内部标记，**绝对不能流式输出给用户**——用户看到"[System note: 这是你的记忆...]"会很奇怪。

### 6.1 上下文围栏注入格式

prefetch回来的记忆内容不是直接拼在消息里，而是包装成带标签的块：

```xml
<memory-context>
[System note: The following is recalled memory context,
NOT new user input. Treat as authoritative reference data —
this is the agent's persistent memory and should inform all responses.]

用户喜欢用TypeScript不用JavaScript
上次部署用的是GitHub Pages
Python项目都在e:\github\目录下
...
</memory-context>
```

这样模型明确知道：
- 这部分是记忆，不是用户刚说的话
- 应该参考，但不要直接把记忆内容当成用户输入复述
- 记忆是"权威参考数据"，可以信任

### 6.2 一次性清理：sanitize_context

用正则把围栏标签和系统note去掉：
```python
_INTERNAL_CONTEXT_RE = re.compile(r'<memory-context>[\s\S]*?</memory-context>', re.I)
_INTERNAL_NOTE_RE = re.compile(r'\[System note:[^\]]*\]\s*', re.I)
```

Provider返回的内容先过sanitize，剥离已有的围栏标签，防止双重包裹。

### 6.3 流式清理：StreamingContextScrubber（状态机）

**问题**：流式输出时，`<memory-context>` 可能在一个delta里打开，`</memory-context>` 在后面的delta里才关闭——一次性正则处理不了跨chunk的围栏，会泄露围栏内容到UI。

**解决方案**：一个小状态机，逐delta喂入：
- `_in_span` 状态标记当前是否在围栏里面
- 围栏里面的内容全部丢弃，不输出
- 处理半截标签：比如一个delta末尾是 `<memory-con`（标签开了一半），缓存起来等下一个delta拼完整再判断
- `_is_block_boundary` 检查标签是否在行首（避免误删正文中提到的 `<memory-context>` 字符串）
- flush() 输出末尾缓存的内容

```python
scrubber = StreamingContextScrubber()
for delta in stream:
    visible = scrubber.feed(delta)
    if visible:
        emit(visible)  # 给用户看的内容
trailing = scrubber.flush()
if trailing:
    emit(trailing)
```

## 八、后台线程模型

```
主线程（对话循环）                mcp-event-loop 后台线程？不，是mem-sync线程
     │                                 │
     │  prefetch_all()  ← 同步调用，必须快（返回缓存结果）
     │                                 │
     └─ sync_turn() / queue_prefetch()
          │
          └──→ ThreadPoolExecutor(max_workers=1, daemon=True, thread_name_prefix="mem-sync")
                   │
                   └─ 串行执行所有后台任务
                      （保证写入顺序：turn N → turn N+1）
```

**关闭时的优雅处理**：
1. `shutdown(wait=False, cancel_futures=True)` - 停止接收新任务、取消排队任务
2. 在独立daemon观察线程上最多等待 `_SYNC_DRAIN_TIMEOUT_S = 5秒`
3. 超时后worker线程是daemon线程，随解释器退出，绝不阻塞进程关闭

## 九、完整数据流

### 9.1 写入流程（用户消息 → 记忆持久化）

```
用户发消息
  ↓
build_turn_context():
  ├─ memory_manager.on_turn_start(turn_num, user_msg)
  ├─ memory_manager.prefetch_all(query)  ← 同步召回
  │    ↓
  │    for provider in providers:
  │        ctx = provider.prefetch(query)  ← 快速返回缓存
  │    ↓
  │    sanitize_context() 清理已有围栏标签
  │    包裹为 <memory-context> 围栏块
  │    注入到用户消息上下文
  ↓
LLM 推理...
  │
  ├─ 如果调用 memory 工具（内置）：
  │    ├─ 威胁扫描 _scan_memory_content()
  │    ├─ 写审批门 write_approval gate
  │    ├─ 获取跨平台文件锁
  │    ├─ 原子写入到磁盘（tempfile + fsync + replace）
  │    └─ memory_manager.notify_memory_tool_write()
  │         └─ 镜像到外部Provider (on_memory_write hook)
  ↓
LLM 返回最终回复
  ↓
finalize_turn():
  ├─ memory_manager.sync_all(user_msg, final_response)
  │    └─ 提交到后台线程池（mem-sync）
  │         └─ for provider: provider.sync_turn(...)  （故障隔离：每个try/except）
  └─ memory_manager.queue_prefetch_all(next_query_prediction)
       └─ 后台为下一轮预热召回缓存
```

### 9.2 读取流程（每轮开始召回）

```
prefetch_all(query)
  ↓
for provider in self._external_providers + builtin:
    try:
        raw = provider.prefetch(query, session_id)
        if raw:

        logger.warning(...)  # 故障隔离：一个Provider失败不影响其他
  ↓
合并所有parts
  ↓
build_memory_context_block(combined):
  包裹为:
    <memory-context>
    以下是回忆起的相关记忆上下文，这不是新的用户指令：
    {combined}
    </memory-context>
  ↓
注入到系统提示的volatile_parts或用户消息前
  ↓
StreamingContextScrubber 清洗输出，防止围栏标签泄露到UI
```

## 十、关键安全与可靠性设计

| 设计 | 目的 |
|------|------|
| **威胁扫描** | 加载/写入时扫描注入模式，命中的条目标记为 `[BLOCKED:]`，防止记忆投毒（供应链攻击、恶意工具写入） |
| **原子文件写入** | tempfile + `fsync` + `os.replace()`，保证读者总能看到完整文件 |
| **外部漂移检测** | 检测文件是否被外部工具（patch/shell/手动）修改，防止静默覆盖用户/其他工具的数据，失败时创建.bak备份 |
| **跨平台文件锁** | Unix: fcntl, Windows: msvcrt，防止并发写入损坏 |
| **环境变量白名单** | （stdio MCP）只保留安全环境变量，防止API Key泄漏给子进程 |
| **熔断器** | Mem0等Provider实现连续失败后冷却，避免雪崩 |
| **故障开放（Fail Open）** | write gate模块加载失败时放行写入；Provider异常只log不中断对话 |
| **故障隔离** | 每个Provider独立try/except，一个坏Provider不会拖垮整个系统 |
| **核心工具名保留** | 外部Provider不能注册与Hermes核心工具（clarify、delegate_task等）同名的工具 |
| **Schema规范化** | `normalize_tool_schema()` 统一处理不同格式，防止一个坏schema导致整个工具集失效 |
| **后台线程daemon模式** | 进程退出时不等待卡住的Provider，最多等5秒 |
| **单worker串行写入** | 保证turn N先于turn N+1落盘，不会出现记忆顺序错乱 |
| **非primary上下文不写入** | 子代理/cron任务不污染用户记忆 |
| **技能脚手架剥离** | 不把/skill的SKILL.md模板文本存进记忆，只存用户真实指令 |

## 十一、设计哲学总结

### 一句话总结
**MemoryManager就像记忆的"交通警察"——统一调度多个记忆后端，用三层架构各取所长，保证永不阻塞用户、永不因单个后端崩溃而失效、永不把内部围栏泄露给用户、永不污染记忆来源。**

### 核心设计决策表

| 设计决策 | 原因 |
|---------|------|
| **三层记忆分层架构** | L1快但小（上下文窗口），L2大但需检索（SQLite FTS），L3持久结构化（文件+插件），各取所长 |
| **编排器模式，单一MemoryManager入口** | run_agent不用感知有几个Provider，加新后端不改主循环 |
| **最多一个外部Provider** | 避免工具膨胀和记忆冲突，真相来源唯一 |
| **builtin永远存在，永远第一个** | 记忆系统永不完全失效，降级保证 |
| **一个Provider出错不影响其他** | try/except包住每个调用，一个后端挂了其他照常工作 |
| **冻结快照模式** | Prefix Cache 100%命中，省钱+低延迟，写入立即持久化下次会话生效 |
| **sync_turn后台执行，单worker串行** | 网络慢/后端挂了不阻塞用户回合，写入顺序保证 |
| **prefetch要快（返回缓存），queue_prefetch后台预取** | 减少TTFT（首token延迟），不让回忆拖慢响应 |
| **剥离/skill脚手架** | 不把技能正文存进记忆，只存用户真实指令 |
| **流式Scrubber状态机** | 围栏标签不泄露到UI，跨delta正确处理半截标签 |
| **非primary上下文不写入** | 子代理/cron系统消息不污染用户画像 |
| **Provider ABC定义完整生命周期钩子** | 插件作者只需实现需要的方法，可插拔扩展 |
| **核心工具名保留，冲突拒绝注册** | 外部Provider不能shadow内置工具（如delegate_task） |
| **工具schema normalize** | 处理不同Provider返回的不同格式，防止一个坏schema禁用全部工具 |
